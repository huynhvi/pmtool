# Requirement – Issue Tracking Dashboard (Input-Agnostic Version)

## 1. Scope
Build an Issue Tracking Dashboard module for PM Tool.

The module must:
- read issue data from an uploaded Excel file
- read data only from sheet "IssuesLog"
- identify actual issue records from real data rows only
- generate a raw data snapshot
- create dashboard views from the generated raw data
- compare the latest 5 snapshots
- provide objective trend evaluation based on percentage change

---

## 2. Input Source

### 2.1 Source File
- Input format: Excel (.xlsx)
- The system must not depend on a fixed file template, fixed file name, fixed total row count, or fixed file formatting.
- The system must work as long as the uploaded Excel file contains sheet "IssuesLog" and the required business fields can be identified from that sheet.

### 2.2 Source Sheet
- The system must read issue data from sheet:
  - "IssuesLog"

### 2.3 Input-Agnostic Principle (Critical)
The system must not depend on:
- predefined input row range
- predefined last row number
- fixed number of formatted rows
- hardcoded cell addresses for the full dataset range
- blank but formatted rows being treated as records

The system must dynamically detect the actual data range from sheet "IssuesLog".

---

## 3. Actual Record Detection Rule (Critical)

### 3.1 Business Rule
A row is counted as an actual issue record only when it contains real issue data.

### 3.2 Record Counting Principle
The system must count only rows that represent real issues.

The system must exclude:
- fully blank rows
- rows that contain formatting only
- rows that are pre-created but have no real issue data
- trailing empty rows below the actual dataset

### 3.3 Recommended Record Validation Rule
A row should be treated as a valid issue record when:
- the row contains a non-empty issue identifier
- and/or the row contains the minimum required fields needed to identify one actual issue item

At minimum, the implementation must ensure that blank rows are never counted as issue records.

### 3.4 Counting Output
Total Issues must be based on the number of actual issue records only.

---

## 4. Required Business Fields
The system must extract and normalize the business fields required for aggregation from sheet "IssuesLog".

At minimum, the dashboard logic must be able to derive these fields:
- Issue ID
- Status
- Severity / Priority
- Issue Type

If the source file uses different header wording but the business meaning is the same, the system may map them through configurable field mapping.

Example:
- Severity may come from "Priority"
- Status may come from another status-related column if that is the actual business status used in the file

The logic must focus on business meaning, not rigid header text only.

---

## 5. Processing Flow
The system must follow this process:

1. Read sheet "IssuesLog"
2. Detect the actual data range dynamically
3. Identify actual issue records only
4. Extract and normalize required business fields
5. Aggregate the data into a raw data snapshot file
6. Save the snapshot file
7. Use the snapshot data as the source for dashboard generation
8. Compare the latest 5 snapshot files for history tracking

---

## 6. Raw Data Snapshot Requirement

### 6.1 Snapshot File Name Format
The generated raw data snapshot file must follow this naming convention:

- Rawdata_yyyymmdd_hhmmss

Example:
- Rawdata_20260422_103000

### 6.2 Purpose
The snapshot file is the standardized data source used for:
- dashboard rendering
- historical comparison
- trend evaluation

### 6.3 Snapshot Output Principle
The raw data snapshot is an internal standardized output generated by the system.

The output structure may be designed by the implementation for dashboard consumption, but the input-reading logic must not depend on any fixed input schema beyond:
- sheet name = "IssuesLog"
- ability to identify required business fields
- ability to detect actual records correctly

---

## 7. Raw Data Snapshot Content
Each generated snapshot must contain the following aggregated metrics:

### 7.1 Total Issues
- Total number of actual issue records

### 7.2 Issue Status Summary
For each status below, the system must store:
- issue count
- percentage based on total number of issues

Statuses:
- Open
- Reopen
- Closed
- To Confirm

### 7.3 Severity Summary for Open Issues
For each severity below, the system must store:
- issue count
- percentage based on total number of issues

Condition:
- Status = Open

Severities:
- High
- Medium
- Low

### 7.4 Severity Summary for Reopen Issues
For each severity below, the system must store:
- issue count
- percentage based on total number of issues

Condition:
- Status = Reopen

Severities:
- High
- Medium
- Low

### 7.5 Issue Type Summary for Open + Reopen
For each issue type below, the system must store:
- issue count
- percentage based on total number of issues

Condition:
- Status = Open OR Reopen

Types:
- Bug
- Configuration
- Improvement

---

## 8. Dashboard Data Source
The dashboard must use the generated raw data snapshot as its data source.

The dashboard must NOT directly calculate metrics from the original uploaded file after snapshot generation.

---

## 9. Dashboard Output

### 9.1 KPI Summary
Display at least:
- Total Issues
- Open
- Reopen
- Closed
- To Confirm

### 9.2 Status Summary
Display issue count and percentage for:
- Open
- Reopen
- Closed
- To Confirm

### 9.3 Severity Summary
Display issue count and percentage for:
- Open → High / Medium / Low
- Reopen → High / Medium / Low

### 9.4 Issue Type Summary
Display issue count and percentage for:
- Bug
- Configuration
- Improvement

Condition:
- based on Open + Reopen only

---

## 10. History Tracking (Critical Requirement)

### 10.1 Scope
The dashboard must compare the latest 5 generated raw data snapshots.

### 10.2 Comparison Target
The system must show increase / decrease trend for at least:
- Total Issues
- Open
- Reopen
- Closed
- To Confirm

The system should also support trend comparison for:
- Open severity distribution
- Reopen severity distribution
- Open + Reopen type distribution

### 10.3 Calculation Rule
Trend must be recorded by percentage change.

Formula:
- % Change = (Current Snapshot - Previous Snapshot) / Previous Snapshot

### 10.4 Display Requirement
History view must:
- compare the latest 5 snapshots
- show increase / decrease direction
- display percentage change clearly

Example:
- Open increased by 12.50%
- Reopen decreased by 8.33%

---

## 11. Objective Evaluation Requirement
The dashboard must provide objective evaluation together with the trend.

### 11.1 Evaluation Principle
Evaluation must be based on percentage change, not on subjective comments only.

### 11.2 Evaluation Examples
- If Open increases → risk is increasing
- If Reopen increases → quality risk is increasing
- If Open decreases → issue backlog is improving
- If Reopen decreases → quality stability is improving

### 11.3 Output Style
Evaluation text should be:
- short
- clear
- objective
- business-friendly

Example:
- "Open issues increased by 12.50% compared to the previous snapshot, indicating rising backlog risk."
- "Reopen issues decreased by 10.00%, showing improvement in issue resolution quality."

---

## 12. Data Normalization Rules
The system must normalize values before aggregation, including:
- status values
- severity values
- issue type values

Examples:
- OPEN → Open
- reopen → Reopen
- HIGH → High

The normalization logic should support configurable mapping so that equivalent source values can be standardized before aggregation.

---

## 13. Error Handling
If required data cannot be extracted from sheet "IssuesLog", the system must:
- stop snapshot generation
- flag missing or invalid fields
- avoid creating misleading dashboard data

The system must clearly distinguish between:
- no actual records found
- required business fields not found
- invalid source data
- sheet "IssuesLog" not found

---

## 14. Design Principle
- input-agnostic processing
- snapshot-based processing
- dynamic record detection
- clear and auditable metric generation
- objective trend evaluation
- simple and readable dashboard output

---

## 15. Key Clarification for Implementation
To avoid parsing failure, the implementation must follow this rule:

- The input file is not guaranteed to have a clean bounded table range.
- The system must read sheet "IssuesLog" and detect actual issue rows from real data only.
- Any blank rows, placeholder rows, or formatted-but-empty rows must be ignored.
- Total issue count and all aggregations must be based only on valid actual records.
